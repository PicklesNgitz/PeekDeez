Even Realities G2 → Deezer Controller

Context

Goal: an Even Realities G2 app that controls the running Deezer Android app on the user's phone. Glasses display now-playing (title / artist / album), transport controls (play/pause/next/previous/seek forward/rewind), a toggle to enable/disable shuffle, optional lyrics view (when available), and the ability to choose a track/album/playlist/music and launch it in Deezer.

Voice is out of scope for this build (user removed it); STT/Soniox/the asr template's mic pipeline are not used. Shuffle toggle is via the G2 touchpad UI only.

Why a 2-component build (confirmed blocker)

The official Even Hub SDK is a WebView hosted by the Even Realities Android app. Its documented Device APIs expose: mic, location, camera, album (single image), IMU, touchpad events, device/user info, local storage, and networking (fetch/XHR/WebSockets). It explicitly does NOT expose: Android MediaSession, now-playing metadata, a notification listener, media-key dispatch, or intents / deezer:// launching (source: hub.evenrealities.com/docs/build/device-apis and /build/networking).

Therefore the G2 plugin alone cannot read what Deezer is playing or command it. We need a second, native Android helper app that uses NotificationListenerService + MediaSessionManager to observe and drive the Deezer app's media session, fires deezer:// intents to launch chosen items, and calls Deezer/lyrics APIs server-side (no browser CORS). The plugin talks to the helper over local HTTP.

- plugin/src/glasses/screens/lyrics.ts — paginated plain lyrics (even-toolkit text pane / 2000-char textContainerUpgrade); scroll advances; double-click back.
- plugin/src/glasses/controller.ts — maps SDK touchpad events (CLICK_EVENT, DOUBLE_CLICK_EVENT, SCROLL_TOP_EVENT, SCROLL_BOTTOM_EVENT) to the active screen's action(). Subscribes to now-playing updates (SSE from helper, falling back to 1 s poll) and re-renders the active screen.
- plugin/src/companion/screens/search.tsx + browse.tsx — phone-side Deezer search + result lists (even-toolkit React components); picking an item calls client.launch(...).

Helper app (new Android Studio / Kotlin project, sideloaded APK)

- AndroidManifest.xml — NotificationListenerService, a foreground Service (keep-alive + battery-opt exemption prompt), INTERNET; intent-filter not required (it launches deezer:// via startActivity, not receives it).
- DeezerMediaListener.kt (NotificationListenerService) — on onNotificationPosted, read StatusBarNotification.extras for the MediaSession parcelable for package com.deezer.android; build a MediaController; register a ControllerCallback for metadata + playback-state + shuffle changes.
- MediaControllerOps.kt — playPause(), next(), previous(), seekDelta(ms), getNowPlaying(), setShuffle(enabled) via transportControls.setShuffleMode(SHUFFLE_MODE_ALL / SHUFFLE_MODE_NONE) and read controller.shuffleMode.
- LocalApiServer.kt (Ktor or NanoHTTPD) bound 127.0.0.1:8765, CORS Access-Control-Allow-Origin: *, JSON endpoints (below).
- DeezerApi.kt (OkHttp) — public API, no auth: /search?q=..., /album/{id}, /playlist/{id}, /genre, mapping Deezer's JSON into the plugin's shape. Calls are server-side so CORS never applies.
- LyricsApi.kt (OkHttp) — https://lrclib.net/api/search?artist_name=&track_name= → plain (and optionally synced LRC) lyrics; 404 → null.
- DeezerLauncher.kt — startActivity(Intent(ACTION_VIEW, Uri.parse("deezer://track|$album|playlist|radio/$id"))); detect Deezer not installed → no-op + flag.

Helper REST protocol (single source of truth for both sides)

GET  /nowplaying          → {title,artist,album,trackId,artUrl,durationMs,positionMs,isPlaying,shuffle:bool,repeat}
GET  /nowplaying/stream   → text/event-stream of the above on change (preferred)
POST /transport           {action:"play"|"pause"|"playPause"|"next"|"previous"|"seekDelta", deltaMs?:120000}
POST /shuffle             {enabled:bool}
GET  /search?q=&type=track|album|playlist → [{id,title,artist,albumName?,artUrl}]
GET  /lyrics?artist=&track= → {plain?:string, synced?:string|null}   (404 if none)
POST /launch              {type:"track"|"album"|"playlist"|"radio", id:string}

Risks (validate early, spike first)

1. Even Hub WebView reaching http://127.0.0.1:8765 — cleartext/localhost in a WebView may be blocked. Spike this on day one with an empty minimal plugin fetching the helper. Fallback chain: 127.0.0.1 → phone LAN IP over http → local HTTPS w/ self-signed (mTLS) only if forced. Log which works.
2. Deezer's MediaSession honoring external commands — play/pause/next/prev/seek and shuffle should be supported; confirm with a 30-line helper PoC before building UI. If a control is unsupported, hide that glasses row rather than ship a dead button.
3. Notification listener being killed — foreground service + battery exemption needed; document the user-facing toggles.
4. Lyrics coverage — partial via lrclib; show "Lyrics unavailable" gracefully, never error the screen.

Out of scope (defer)

- Voice / STT / Soniox / g2-microphone (user removed voice).
- Deezer OAuth / user library / personal playlists (Phase 1 = public search only).
- Synced/karaoke lyrics highlighting (Phase 1 = plain paginated).
- Volume control (MediaSession can't reliably set Deezer's volume).
